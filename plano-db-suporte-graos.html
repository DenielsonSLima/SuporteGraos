⚠ Causa Raiz — Por que "ajusta um, quebra outro"

Cada módulo faz suas próprias queries e cálculos no frontend de forma independente e inconsistente. Quando você muda a lógica de um módulo, os outros módulos continuam calculando do jeito antigo. Não existe uma fonte única de verdade (single source of truth) no banco. Os totais de cada tela são calculados de formas diferentes para os mesmos dados. A solução definitiva é mover 100% dos cálculos para o PostgreSQL via Views SQL e RPCs, e o frontend apenas lê e exibe — nunca calcula.

02 · Princípios

As regras que nunca podem ser quebradas

✅ Frontend DEVE
Chamar RPCs para toda escrita
Ler dados de Views (nunca de tabelas raw)
Invalidar queryKeys exatas após cada mutação
Tratar loading e erro de cada query separadamente
Passar IDs e valores brutos para o RPC
❌ Frontend NUNCA
Calcular totais, médias, saldos
Fazer múltiplas queries e somar no JS
Inserir em tabelas diretamente (sem RPC)
Gerar títulos financeiros no frontend
Calcular "Contas a Pagar" ou "Resultado" localmente
✅ Banco DEVE
Calcular 100% dos totais em Views SQL
Gerar títulos financeiros via Triggers
Garantir integridade via FOREIGN KEYS + transactions
Retornar todos os dados necessários por tela em 1 RPC
Validar regras de negócio (ex: não pagar mais que o total)
❌ Banco NUNCA
Armazenar campos calculados em tabelas (sem trigger de sync)
Permitir delete em cascata sem verificação de saldo
Ter RPCs sem transações (BEGIN/COMMIT)
Ter Views sem índices nas colunas de JOIN

05 · Views SQL

Cálculos centralizados — uma vez, para todos os módulos

Regra: Se dois módulos precisam do mesmo número, ambos leem da mesma View. Nunca duplicar lógica de cálculo.

VIEW: v_resumo_pedido_compra

Usada pelo módulo Ped. Compra (cards de totais, lista, detalhe) e pelo Financeiro (dívida do fornecedor).

SQL
CREATE OR REPLACE VIEW v_resumo_pedido_compra AS
SELECT
  pc.id,
  pc.numero,
  pc.parceiro_id,
  par.nome                                          AS parceiro_nome,
  par.nome_fantasia                                 AS parceiro_fantasia,
  pc.socio_id,
  pc.produto,
  pc.data_pedido,
  pc.preco_sc,
  pc.qtd_contrato_sc,
  pc.status,

  -- Físico: sacas já carregadas (soma dos romaneios)
  COALESCE(SUM(r.peso_origem_kg) / NULLIF(pc.AVG_kg_por_sc, 0), 0) 
                                                    AS qtd_carregada_sc,
  COALESCE(SUM(r.peso_origem_kg), 0)               AS peso_carregado_kg,

  -- Físico: sacas entregues (peso destino confirmado)
  COALESCE(SUM(r.peso_destino_kg) / NULLIF(60, 0), 0) 
                                                    AS qtd_entregue_sc,

  -- Saldo físico a carregar
  pc.qtd_contrato_sc - 
    COALESCE(SUM(r.peso_origem_kg) / 60, 0)         AS saldo_carregar_sc,

  -- Financeiro: valor total do contrato
  pc.qtd_contrato_sc * pc.preco_sc                 AS valor_contrato,

  -- Dívida: soma de títulos a pagar para este pedido
  COALESCE(tp_agg.valor_original_total, 0)          AS valor_divida_total,
  COALESCE(tp_agg.valor_pago_total, 0)              AS valor_pago_total,
  COALESCE(tp_agg.valor_original_total, 0) - 
    COALESCE(tp_agg.valor_pago_total, 0)             AS saldo_pendente,

  -- Mercadoria em trânsito (carregado mas não entregue)
  COALESCE(transito.valor_transito, 0)              AS valor_em_transito

FROM pedidos_compra pc
LEFT JOIN parceiros par ON par.id = pc.parceiro_id
LEFT JOIN romaneios r ON r.pedido_compra_id = pc.id 
  AND r.status != 'cancelado'

-- Agrega títulos a pagar deste pedido
LEFT JOIN LATERAL (
  SELECT
    SUM(tp.valor_original) AS valor_original_total,
    SUM(tp.valor_pago)     AS valor_pago_total
  FROM titulos_pagar tp
  WHERE tp.pedido_compra_id = pc.id
    AND tp.tipo = 'fornecedor_grao'
) tp_agg ON TRUE

-- Valor em trânsito = romaneios em_transito * preço venda
LEFT JOIN LATERAL (
  SELECT
    SUM((r2.peso_origem_kg / r2.kg_por_sc) * pv.preco_sc) AS valor_transito
  FROM romaneios r2
  JOIN pedidos_venda pv ON pv.id = r2.pedido_venda_id
  WHERE r2.pedido_compra_id = pc.id
    AND r2.status = 'em_transito'
) transito ON TRUE

GROUP BY pc.id, par.nome, par.nome_fantasia, tp_agg.valor_original_total, 
         tp_agg.valor_pago_total, transito.valor_transito;

VIEW: v_resumo_pedido_venda

Usada pelo módulo Ped. Venda (cards, lista, performance do produto).

SQL
CREATE OR REPLACE VIEW v_resumo_pedido_venda AS
SELECT
  pv.id,
  pv.numero,
  pv.parceiro_id,
  par.nome                                          AS parceiro_nome,
  pv.produto,
  pv.data_pedido,
  pv.preco_sc,
  pv.qtd_contrato_sc,
  pv.status,

  -- Quantidade carregada (romaneios vinculados)
  COALESCE(SUM(r.peso_origem_kg) / 60, 0)         AS qtd_carregada_sc,

  -- Quantidade entregue (peso destino confirmado)
  COALESCE(SUM(r.peso_destino_kg) / 60, 0)        AS qtd_entregue_sc,

  -- Valor de faturamento contratado
  pv.qtd_contrato_sc * pv.preco_sc                 AS faturamento_contratado,

  -- Faturamento real (ref. peso destino)
  COALESCE(
    SUM(r.peso_destino_kg / 60 * pv.preco_sc)
    FILTER (WHERE r.peso_destino_kg IS NOT NULL), 0
  )                                                 AS faturamento_real,

  -- Custo grão (pelo peso origem carregado)
  COALESCE(
    SUM((r.peso_origem_kg / r.kg_por_sc) * pc.preco_sc), 0
  )                                                 AS custo_grao_total,

  -- Custo frete (logística vinculada)
  COALESCE(SUM(l.custo_frete_total), 0)            AS custo_frete_total,

  -- Total custos
  COALESCE(SUM((r.peso_origem_kg / r.kg_por_sc) * pc.preco_sc), 0) +
    COALESCE(SUM(l.custo_frete_total), 0)          AS custo_total,

  -- Lucro bruto projetado (ref. peso origem)
  (COALESCE(SUM(r.peso_origem_kg / r.kg_por_sc * pv.preco_sc), 0)) -
  (COALESCE(SUM((r.peso_origem_kg / r.kg_por_sc) * pc.preco_sc), 0)) -
  (COALESCE(SUM(l.custo_frete_total), 0))           AS lucro_bruto,

  -- Saldo a receber (título a receber pendente)
  COALESCE(tr_agg.saldo_receber, 0)                  AS saldo_receber,

  -- Carga em trânsito (valor ref. preço venda)
  COALESCE(
    SUM(r.peso_origem_kg / 60 * pv.preco_sc)
    FILTER (WHERE r.status = 'em_transito'), 0
  )                                                 AS carga_em_transito

FROM pedidos_venda pv
LEFT JOIN parceiros par ON par.id = pv.parceiro_id
LEFT JOIN romaneios r ON r.pedido_venda_id = pv.id 
  AND r.status != 'cancelado'
LEFT JOIN logistica l ON l.id = r.logistica_id
LEFT JOIN pedidos_compra pc ON pc.id = r.pedido_compra_id
LEFT JOIN LATERAL (
  SELECT SUM(tr.valor_original - tr.valor_recebido) AS saldo_receber
  FROM titulos_receber tr
  WHERE tr.pedido_venda_id = pv.id AND tr.status != 'quitado'
) tr_agg ON TRUE

GROUP BY pv.id, par.nome, tr_agg.saldo_receber;

VIEW: v_posicao_caixa

Usada pelo Caixa, Dashboard e Financeiro. Esta é a fonte única de todos os saldos patrimoniais.

SQL
CREATE OR REPLACE VIEW v_posicao_caixa AS
SELECT
  -- ATIVOS
  COALESCE(saldos_banco.total, 0)                  AS saldo_disponivel,
  COALESCE(transito.valor, 0)                      AS mercadoria_em_transito,
  COALESCE(a_receber.valor, 0)                     AS contas_a_receber,
  COALESCE(patrimonio.valor, 0)                    AS patrimonio_bens,

  COALESCE(saldos_banco.total, 0) +
  COALESCE(transito.valor, 0) +
  COALESCE(a_receber.valor, 0) +
  COALESCE(patrimonio.valor, 0)                    AS total_ativos,

  -- PASSIVOS
  COALESCE(a_pagar_fornecedores.valor, 0)          AS fornecedores_graos,
  COALESCE(a_pagar_fretes.valor, 0)               AS fretes_a_pagar,
  COALESCE(a_pagar_outros.valor, 0)               AS outros_a_pagar,

  COALESCE(a_pagar_fornecedores.valor, 0) +
  COALESCE(a_pagar_fretes.valor, 0) +
  COALESCE(a_pagar_outros.valor, 0)               AS total_passivos,

  -- PATRIMÔNIO LÍQUIDO
  (
    COALESCE(saldos_banco.total, 0) +
    COALESCE(transito.valor, 0) +
    COALESCE(a_receber.valor, 0) +
    COALESCE(patrimonio.valor, 0)
  ) - (
    COALESCE(a_pagar_fornecedores.valor, 0) +
    COALESCE(a_pagar_fretes.valor, 0) +
    COALESCE(a_pagar_outros.valor, 0)
  )                                                 AS patrimonio_liquido

FROM
  (SELECT SUM(saldo_atual) AS total FROM contas_bancarias) saldos_banco,

  (SELECT SUM(r.peso_origem_kg / r.kg_por_sc * pv.preco_sc) AS valor
   FROM romaneios r
   JOIN pedidos_venda pv ON pv.id = r.pedido_venda_id
   WHERE r.status = 'em_transito') transito,

  (SELECT SUM(valor_original - valor_recebido) AS valor
   FROM titulos_receber WHERE status != 'quitado') a_receber,

  (SELECT COALESCE(SUM(valor_atual), 0) AS valor 
   FROM patrimonio_bens WHERE status = 'ativo') patrimonio,

  (SELECT SUM(valor_original - valor_pago) AS valor
   FROM titulos_pagar WHERE tipo = 'fornecedor_grao' 
   AND status != 'quitado') a_pagar_fornecedores,

  (SELECT SUM(valor_original - valor_pago) AS valor
   FROM titulos_pagar WHERE tipo = 'frete' 
   AND status != 'quitado') a_pagar_fretes,

  (SELECT SUM(valor_original - valor_pago) AS valor
   FROM titulos_pagar WHERE tipo NOT IN ('fornecedor_grao', 'frete') 
   AND status != 'quitado') a_pagar_outros;
06 · RPCs Atômicos

Toda escrita passa por aqui — com transação

RPC: criar_romaneio (o mais crítico)

Cria o romaneio E automaticamente gera os títulos a pagar (fornecedor + frete) em uma única transação. Se qualquer parte falhar, tudo faz rollback.

SQL
CREATE OR REPLACE FUNCTION criar_romaneio(
  p_pedido_compra_id  uuid,
  p_pedido_venda_id   uuid,
  p_transportadora_id uuid,
  p_motorista_nome    text,
  p_placa_veiculo     text,
  p_data_carga        date,
  p_peso_origem_kg    numeric,
  p_custo_frete_total numeric,
  p_quem_paga_frete   text,  -- 'empresa' | 'produtor'
  p_kg_por_sc         numeric DEFAULT 60
) RETURNS json AS $$
DECLARE
  v_logistica_id    uuid;
  v_romaneio_id     uuid;
  v_pc              RECORD;
  v_qtd_sc          numeric;
  v_valor_grao      numeric;
  v_titulo_forn_id  uuid;
  v_titulo_frete_id uuid;
BEGIN

  -- 1. Busca dados do pedido de compra
  SELECT * INTO v_pc FROM pedidos_compra WHERE id = p_pedido_compra_id;
  IF NOT FOUND THEN
    RAISE EXCEPTION 'Pedido de compra não encontrado';
  END IF;

  -- 2. Calcula sacas e valor
  v_qtd_sc     := p_peso_origem_kg / p_kg_por_sc;
  v_valor_grao := v_qtd_sc * v_pc.preco_sc;

  -- 3. Cria registro de logística
  INSERT INTO logistica (
    transportadora_id, motorista_nome, placa_veiculo,
    custo_frete_total, quem_paga_frete
  ) VALUES (
    p_transportadora_id, p_motorista_nome, p_placa_veiculo,
    p_custo_frete_total, p_quem_paga_frete
  ) RETURNING id INTO v_logistica_id;

  -- 4. Cria o romaneio
  INSERT INTO romaneios (
    pedido_compra_id, pedido_venda_id, logistica_id,
    data_carga, peso_origem_kg, kg_por_sc, status
  ) VALUES (
    p_pedido_compra_id, p_pedido_venda_id, v_logistica_id,
    p_data_carga, p_peso_origem_kg, p_kg_por_sc, 'em_transito'
  ) RETURNING id INTO v_romaneio_id;

  -- 5. Gera título a pagar para o fornecedor (grão)
  INSERT INTO titulos_pagar (
    tipo, parceiro_id, pedido_compra_id, romaneio_id,
    valor_original, valor_pago, status, data_vencimento, descricao
  ) VALUES (
    'fornecedor_grao',
    v_pc.parceiro_id,
    p_pedido_compra_id,
    v_romaneio_id,
    v_valor_grao,
    0,
    'aberto',
    CURRENT_DATE + INTERVAL '30 days',
    'Grão - ' || v_pc.produto || ' - ROM:' || v_romaneio_id
  ) RETURNING id INTO v_titulo_forn_id;

  -- 6. Gera título a pagar para o frete (se empresa paga)
  IF p_quem_paga_frete = 'empresa' AND p_custo_frete_total > 0 THEN
    INSERT INTO titulos_pagar (
      tipo, parceiro_id, pedido_compra_id, romaneio_id,
      valor_original, valor_pago, status, data_vencimento, descricao
    ) VALUES (
      'frete',
      p_transportadora_id,
      p_pedido_compra_id,
      v_romaneio_id,
      p_custo_frete_total,
      0,
      'aberto',
      CURRENT_DATE + INTERVAL '15 days',
      'Frete - ROM:' || v_romaneio_id
    ) RETURNING id INTO v_titulo_frete_id;
  END IF;

  -- 7. Retorna resultado
  RETURN json_build_object(
    'success', true,
    'romaneio_id', v_romaneio_id,
    'logistica_id', v_logistica_id,
    'titulo_fornecedor_id', v_titulo_forn_id,
    'titulo_frete_id', v_titulo_frete_id,
    'valor_grao', v_valor_grao,
    'qtd_sc', v_qtd_sc
  );

EXCEPTION WHEN OTHERS THEN
  RAISE EXCEPTION 'criar_romaneio falhou: %', SQLERRM;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

RPC: confirmar_peso_destino

Confirma a entrega — atualiza o romaneio com o peso real de destino e gera o título a receber automaticamente.

SQL
CREATE OR REPLACE FUNCTION confirmar_peso_destino(
  p_romaneio_id    uuid,
  p_peso_destino_kg numeric
) RETURNS json AS $$
DECLARE
  v_rom     RECORD;
  v_pv      RECORD;
  v_qtd_sc  numeric;
  v_valor   numeric;
BEGIN
  SELECT r.*, l.custo_frete_total
  INTO v_rom
  FROM romaneios r
  JOIN logistica l ON l.id = r.logistica_id
  WHERE r.id = p_romaneio_id;

  SELECT * INTO v_pv FROM pedidos_venda WHERE id = v_rom.pedido_venda_id;

  v_qtd_sc := p_peso_destino_kg / v_rom.kg_por_sc;
  v_valor   := v_qtd_sc * v_pv.preco_sc;

  -- Atualiza romaneio com peso destino e muda status
  UPDATE romaneios
  SET peso_destino_kg = p_peso_destino_kg,
      status          = 'entregue'
  WHERE id = p_romaneio_id;

  -- Gera título a receber do comprador
  INSERT INTO titulos_receber (
    tipo, parceiro_id, pedido_venda_id, romaneio_id,
    valor_original, valor_recebido, status, data_vencimento, descricao
  ) VALUES (
    'venda_grao',
    v_pv.parceiro_id,
    v_rom.pedido_venda_id,
    p_romaneio_id,
    v_valor, 0, 'aberto',
    CURRENT_DATE + INTERVAL '30 days',
    'Venda grão - ROM:' || p_romaneio_id
  );

  RETURN json_build_object(
    'success', true,
    'qtd_sc_destino', v_qtd_sc,
    'valor_a_receber', v_valor
  );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

RPC: lancar_pagamento_titulo

Único RPC para todos os pagamentos (fornecedor e frete). Atualiza valor_pago e status automaticamente.

SQL
CREATE OR REPLACE FUNCTION lancar_pagamento_titulo(
  p_titulo_id         uuid,
  p_valor             numeric,
  p_data_pagamento    date,
  p_tipo              text,  -- 'adiantamento' | 'liquidacao_parcial' | 'liquidacao_total'
  p_conta_bancaria_id uuid  DEFAULT NULL,
  p_observacao        text  DEFAULT NULL
) RETURNS json AS $$
DECLARE
  v_titulo  RECORD;
  v_novo_pago numeric;
  v_status    text;
BEGIN
  SELECT * INTO v_titulo 
  FROM titulos_pagar WHERE id = p_titulo_id FOR UPDATE;

  IF v_titulo.status = 'quitado' THEN
    RAISE EXCEPTION 'Título já está quitado';
  END IF;

  v_novo_pago := v_titulo.valor_pago + p_valor;

  IF v_novo_pago > v_titulo.valor_original THEN
    RAISE EXCEPTION 'Pagamento excede valor original do título';
  END IF;

  v_status := CASE
    WHEN v_novo_pago >= v_titulo.valor_original THEN 'quitado'
    WHEN v_novo_pago > 0                        THEN 'parcial'
    ELSE 'aberto'
  END;

  -- Atualiza o título
  UPDATE titulos_pagar
  SET valor_pago = v_novo_pago,
      status     = v_status
  WHERE id = p_titulo_id;

  -- Registra o movimento
  INSERT INTO pagamentos_titulos_pagar (
    titulo_id, valor, tipo, data_pagamento, 
    conta_bancaria_id, observacao
  ) VALUES (
    p_titulo_id, p_valor, p_tipo, p_data_pagamento,
    p_conta_bancaria_id, p_observacao
  );

  -- Debita conta bancária se informada
  IF p_conta_bancaria_id IS NOT NULL THEN
    UPDATE contas_bancarias
    SET saldo_atual = saldo_atual - p_valor
    WHERE id = p_conta_bancaria_id;
  END IF;

  RETURN json_build_object(
    'success', true,
    'novo_status', v_status,
    'valor_pago_total', v_novo_pago,
    'saldo_restante', v_titulo.valor_original - v_novo_pago
  );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

Mapa de invalidação — o que invalidar após cada RPC

Cada mutação no frontend deve chamar queryClient.invalidateQueries() para as keys exatas abaixo.

// hooks/use-criar-romaneio.ts
export function useCriarRomaneio() {
  const queryClient = useQueryClient()

  return useMutation({
    mutationFn: async (data: CriarRomaneioInput) => {
      const { data: result, error } = await supabase
        .rpc('criar_romaneio', {
          p_pedido_compra_id:  data.pedidoCompraId,
          p_pedido_venda_id:   data.pedidoVendaId,
          p_transportadora_id: data.transportadoraId,
          p_motorista_nome:    data.motoristaNome,
          p_placa_veiculo:     data.placaVeiculo,
          p_data_carga:        data.dataCarga,
          p_peso_origem_kg:    data.pesoOrigemKg,
          p_custo_frete_total: data.custoFrete,
          p_quem_paga_frete:   data.quemPagaFrete,
        })

      if (error) throw error
      return result
    },

    // Invalida TUDO que foi afetado por esta operação
    onSuccess: (_, variables) => {
      queryClient.invalidateQueries({ queryKey: ['romaneios'] })
      queryClient.invalidateQueries({ queryKey: ['pedidos_compra', variables.pedidoCompraId] })
      queryClient.invalidateQueries({ queryKey: ['pedidos_venda', variables.pedidoVendaId] })
      queryClient.invalidateQueries({ queryKey: ['titulos_pagar'] })
      queryClient.invalidateQueries({ queryKey: ['caixa'] })
      queryClient.invalidateQueries({ queryKey: ['dashboard'] })
    },
  })
}

08 · Ordem de Implementação

Como migrar sem quebrar nada

Estratégia: Construir as Views e RPCs em paralelo com o código atual. Migrar módulo por módulo, sempre validando que os números batem antes de remover a lógica antiga do frontend.
1
Criar as tabelas auxiliares que faltam
Verificar se titulos_pagar, titulos_receber, pagamentos_titulos_pagar existem com a estrutura correta. Criar migrations do zero se necessário. NÃO apagar dados existentes.
~2h
2
Migrar dados históricos para a nova estrutura
Rodar um script SQL que gera os titulos_pagar retroativos para todos os romaneios existentes que não têm título vinculado. Esse é o passo que vai corrigir o Financeiro → Fornecedores que mostrava R$0,00.
~3h
3
Criar as Views SQL
Criar v_resumo_pedido_compra, v_resumo_pedido_venda, v_posicao_caixa e v_dashboard. Testar cada View com os dados existentes e validar que os números batem com o que a tela mostra hoje.
~4h
4
Criar os RPCs atômicos
Criar criar_romaneio, confirmar_peso_destino, lancar_pagamento_titulo, lancar_recebimento_venda. Testar cada RPC isoladamente via Supabase Dashboard SQL Editor.
~6h
5
Migrar módulo Caixa primeiro
O Caixa é o mais simples pois é só leitura — trocar para ler de v_posicao_caixa via uma única query. Validar que os números batem. Primeiro módulo a ficar 100% correto.
~2h
6
Migrar Ped. Compra (leitura)
Trocar as queries do módulo Ped. Compra para ler de v_resumo_pedido_compra. Cards de totais, lista e detalhe passam a usar os campos da View. Verificar: "Total Comprado" passa a mostrar o valor correto.
~3h
7
Migrar Ped. Venda (leitura + performance)
Trocar queries para v_resumo_pedido_venda. A seção "Performance do Produto" passa a mostrar Custo Grão, Custo Frete e Lucro Bruto corretos vindos da View, não calculados no frontend.
~3h
8
Migrar Financeiro (Contas a Pagar e Receber)
Módulo Financeiro passa a ler de titulos_pagar e titulos_receber (estas são tabelas normalizadas, não Views). Dívida Consolidada do ARMAZEM ROCHA vai aparecer corretamente porque agora existem os títulos.
~4h
9
Migrar escritas para RPCs + invalidação TanStack
Trocar todos os INSERTs diretos (romaneio, pagamento, recebimento) para chamar os RPCs. Implementar o mapa de invalidação de queryKeys. A partir deste ponto: alterar qualquer dado propaga automaticamente para todos os módulos.
~6h
10
Validação final e remoção do código antigo
Testar o fluxo completo: criar pedido compra → criar romaneio → verificar Financeiro → pagar → verificar Caixa → confirmar entrega → verificar Contas a Receber → receber → verificar Dashboard. Remover todo código de cálculo do frontend.
~4h
